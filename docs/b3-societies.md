# B3 â€” Societies listing and detail pages

The Societies section comprises two routes: a listing page (`/societies`) showing all 21 affiliated
societies with province filter and free-text search, and individual detail pages (`/societies/[slug]`)
statically generated at build time.

## What was built

| Route | Component | Rendering |
|---|---|---|
| `/societies` | `app/(marketing)/societies/page.tsx` | Server Component + client island |
| `/societies/[slug]` | `app/(marketing)/societies/[slug]/page.tsx` | SSG via `generateStaticParams` |
| Filter / search UI | `app/(marketing)/societies/SocietiesClient.tsx` | Client Component (`'use client'`) |
| Society card | `components/societies/SocietyCard.tsx` | Server-safe presentational component |

The listing page fetches all societies server-side, then passes them as a prop to the
`SocietiesClient` island. The island owns all filter and search interactivity â€” the Server Component
itself is static and requires no JavaScript to render initial content.

## Adding a society in Studio

Open the **Society** document type in Sanity Studio and create a new document. Required fields are
`name`, `slug`, `province` (one of the nine SA province codes), and `region` (city or area). Optional
but recommended: `meets` (meeting schedule as a plain string, e.g. "2nd Tuesday, 19:30"), `venue`,
`memberCount` (integer), `founded` (year), `description`, and `website`. Set `markBadge` to true for
societies that run an SAOC awards-mark programme â€” a small award icon appears on their card. Publish
the document and the revalidation webhook flushes the listing cache automatically.

## Province filter and free-text search

Both filters are stored in the URL as query parameters (`?province=GP&q=pretoria`) so the state is
bookmarkable and shareable. `SocietiesClient` reads them with `useSearchParams` and writes them with
`router.replace` (no full navigation, no scroll jump). Selecting a province chip sets `?province=`
to the two-letter code; selecting "All" removes the parameter. The search input matches against
`name` and `region` fields (case-insensitive). Both filters apply together â€” province AND text must
match. Clearing both returns the full list.

## Static fallback

If Sanity is unreachable at build time or returns an empty result, the listing page falls back to
the 21 societies in `lib/data/societies.ts`. The same fallback applies in `generateStaticParams` â€”
all 21 slugs are pre-rendered even without a Sanity connection. Fallback entries carry synthetic
`_id` values (`static-0`, `static-1`, â€¦). As soon as Sanity returns at least one society document,
the static list is replaced entirely â€” the two sources are never merged.

Cache tags: `['society', 'sanity']` on the listing query; `['society', '<slug>', 'sanity']` on each
detail query.

## Slug format

Slugs are auto-generated by Sanity Studio from the society `name` using the built-in slugify helper.
The slugify function lowercases the name, replaces any run of non-alphanumeric characters with a
hyphen, and strips leading/trailing hyphens â€” for example "Orchid Society of Pretoria" becomes
`orchid-society-of-pretoria`. Slugs must be unique across all society documents. Once a slug is
published and indexed, changing it breaks existing bookmarks and inbound links â€” treat published
slugs as permanent.
